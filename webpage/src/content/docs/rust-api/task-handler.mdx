`DurableExecution` is the trait you implement to define a durable workflow. Zart calls `run` each time a worker claims your execution. Completed steps are skipped automatically — only the next pending step executes.

## The Trait

```rust
#[async_trait]
pub trait DurableExecution: Send + Sync {
    type Data:   serde::de::DeserializeOwned + Send + Sync;
    type Output: serde::Serialize + Send + Sync;

    async fn run(
        &self,
        ctx: &mut TaskContext,
        data: Self::Data,
    ) -> Result<Self::Output, TaskError>;

    /// Retry the full execution this many times on task-level failure. Default: 0.
    fn max_retries(&self) -> usize { 0 }

    /// Wall-clock timeout for the entire execution. Default: None.
    fn timeout(&self) -> Option<Duration> { None }
}
```

Both `Data` and `Output` must implement `serde` traits — Zart serializes them to the database.

## TaskRegistry

Register all handlers before starting a worker. The string name is how you reference the workflow when scheduling.

```rust
let mut registry = TaskRegistry::new();
registry.register("onboarding", OnboardingTask);
registry.register("checkout",   CheckoutTask);
```

<Aside type="caution">
Registering two handlers under the same name panics at startup. Names are case-sensitive.
</Aside>

## Core Steps

Steps are defined with `#[zart_step]` or by manually implementing the [`ZartStep`](/rust-api/zart-step/) trait. Each step's result is persisted to the database before the workflow moves on.

### Sequential Steps

Use `ctx.execute_step()` to run a step sequentially:

```rust
let customer_id: String = ctx.execute_step(CreateCustomerStep { email: data.email }).await?;
```

- The step name (from `ZartStep::step_name()`) must be unique within the execution.
- The step return type must implement `Serialize + DeserializeOwned`.
- If a result is already stored for this step, the step is skipped and the cached result is returned.

### Steps with Retry

Override `retry_config()` in your `ZartStep` impl to enable retries:

```rust
struct PlaceOrderStep;

#[async_trait]
impl ZartStep for PlaceOrderStep {
    type Output = String;
    fn step_name(&self) -> Cow<'static, str> { Cow::Borrowed("place-order") }
    fn retry_config(&self) -> Option<RetryConfig> {
        Some(RetryConfig::exponential(5, Duration::from_secs(1)))
    }
    async fn run(&self, ctx: StepContext) -> Result<Self::Output, StepError> {
        payments_api.charge(&data.card).await
    }
}

// Usage:
let order_id: String = ctx.execute_step(PlaceOrderStep).await?;
```

See [ZartStep](/rust-api/zart-step/) for all available configuration options.

## Timing

Use these methods to pause a workflow for a fixed duration or until a specific time. The worker releases the task during the wait — it does not hold a database connection or thread.

### `ctx.sleep()`

```rust
// Pause for 30 minutes, then continue
ctx.sleep(Duration::from_secs(1800)).await?;
```

### `ctx.sleep_until()`

```rust
// Resume at a specific timestamp
ctx.sleep_until(next_billing_date).await?;
```

<Aside type="note">
Both sleep methods are durable. If the process restarts during the sleep, the execution resumes only after the full duration has elapsed — the sleep does not restart.
</Aside>

## Events

`wait_for_event` suspends the workflow until an external signal arrives — a human approval, a webhook callback, or a message from another system.

### `ctx.wait_for_event()`

```rust
// Suspend until "manager-approval" event is delivered, or fail after 24h
let decision: ApprovalPayload = ctx.wait_for_event(
    "manager-approval",
    Some(Duration::from_secs(86_400)),
).await?;
```

Deliver the event from anywhere using `DurableScheduler::offer_event()` or via the HTTP API:

```bash
POST /api/v1/events/{execution_id}/manager-approval
{"approved": true, "reviewer": "bob@example.com"}
```

Returns `Err(TaskError::EventTimeout)` if the timeout elapses before the event arrives. See [Wait for Event](/rust-api/wait-for-event/) for full patterns.

## Capture Values

Capture variables let you durably persist a synchronous, pure value without defining a full step. Ideal for timestamps, environment reads, or generated IDs:

```rust
use zart::zart_capture;

let started_at = zart_capture!(ctx, "started-at", chrono::Utc::now());
let user_tz = zart_capture!(ctx, "user-tz", env::var("TZ").unwrap_or_default());
```

Or use the built-in `now()` helper for timestamps:

```rust
let started_at = ctx.now("started-at").await?;
```

On first run the expression is evaluated and persisted. On replay the cached value is returned — the expression is never re-evaluated. See [Capture Variables](/rust-api/capture-variables/) for full details and when to use captures vs regular steps.

## Parallel Execution

Fan out independent work across multiple concurrent sub-tasks. Each `schedule_step` call creates a separate row in the scheduler — sub-tasks can run on different workers simultaneously.

### `ctx.schedule_step()` / `ctx.wait_all()`

```rust
// Schedule three steps to run in parallel
let h1 = ctx.schedule_step(SendEmailStep { to: user.email });
let h2 = ctx.schedule_step(SetupBillingStep { customer_id });
let h3 = ctx.schedule_step(ProvisionStep { user_id });

// Durably wait for all three — survives restarts
let [email_ok, billing_ok, provision_ok] = ctx.wait_all(vec![h1, h2, h3]).await?;
```

See [Parallel Steps](/rust-api/parallel-steps/) for error handling and dynamic fan-out patterns.

## Metadata

### `ctx.execution_id()`

Returns the current execution's ID string. Useful for building idempotency keys for external API calls that must not be duplicated.

```rust
let key = format!("charge-{}", ctx.execution_id());
stripe.charge_with_idempotency_key(&card, amount, &key).await?;
```

### `ctx.data()` / `ctx.set_data()`

A mutable JSON blob persisted alongside the execution. Use it to accumulate state across steps when you need values that aren't step return values.

```rust
// Read current progress (default if not yet set)
let mut progress: Progress = serde_json::from_value(ctx.data().clone())
    .unwrap_or_default();

progress.items_processed += 1;
ctx.set_data(serde_json::to_value(&progress)?);
```

## RetryConfig

Controls retry behaviour for steps (via `ZartStep::retry_config()`) and the task-level `max_retries`. All three constructors share the same fields: number of attempts, initial delay, and backoff multiplier.

| Constructor | Delay between retries |
|---|---|
| `RetryConfig::none()` | No retries — fail immediately |
| `RetryConfig::fixed(n, delay)` | Constant `delay` between each attempt |
| `RetryConfig::exponential(n, initial)` | Doubles each time: `initial`, `2×`, `4×`, … |

```rust
// No retries (idempotent steps that must not be duplicated)
RetryConfig::none()

// 3 retries, 5 seconds apart
RetryConfig::fixed(3, Duration::from_secs(5))

// 5 retries: 1s, 2s, 4s, 8s, 16s
RetryConfig::exponential(5, Duration::from_secs(1))
```

## Full Example

A five-step onboarding workflow combining steps, retries, and an event wait:

```rust
use zart::prelude::*;
use async_trait::async_trait;
use serde::{Deserialize, Serialize};
use std::time::Duration;

#[derive(Debug, Deserialize)]
pub struct OnboardingData {
    pub user_id: String,
    pub email:   String,
}

#[derive(Debug, Serialize)]
pub struct OnboardingResult {
    pub customer_id: String,
}

// ── Step definitions ─────────────────────────────────────────────────────────

#[zart::zart_step("send-welcome-email")]
async fn send_welcome_email(email: &str, ctx: zart::context::StepContext)
    -> Result<(), zart::error::StepError>
{
    send_welcome_email(email).await
}

#[zart::zart_step("create-stripe-customer", retry = "exponential(3, 2s)")]
async fn create_stripe_customer(email: &str, ctx: zart::context::StepContext)
    -> Result<String, zart::error::StepError>
{
    create_stripe_customer(email).await
}

#[zart::zart_step("provision-resource")]
async fn provision_resource(user_id: &str, customer_id: &str, ctx: zart::context::StepContext)
    -> Result<(), zart::error::StepError>
{
    provision_aws_resources(user_id, customer_id).await
}

#[zart::zart_step("activate-account")]
async fn activate_account(user_id: &str, ctx: zart::context::StepContext)
    -> Result<(), zart::error::StepError>
{
    activate_account(user_id).await
}

// ── Durable handler ──────────────────────────────────────────────────────────

pub struct OnboardingTask;

#[async_trait]
impl DurableExecution for OnboardingTask {
    type Data   = OnboardingData;
    type Output = OnboardingResult;

    async fn run(
        &self,
        ctx: &mut TaskContext,
        data: OnboardingData,
    ) -> Result<Self::Output, TaskError> {
        // Step 1 — send welcome email
        ctx.execute_step(send_welcome_email(&data.email)).await?;

        // Step 2 — create Stripe customer (retry 3× with exponential backoff)
        let customer_id: String = ctx.execute_step(create_stripe_customer(&data.email)).await?;

        // Step 3 — provision cloud resources
        ctx.execute_step(provision_resource(&data.user_id, &customer_id)).await?;

        // Step 4 — wait up to 48h for email verification
        let _: VerificationPayload = ctx.wait_for_event(
            "email-verified",
            Some(Duration::from_secs(172_800)),
        ).await?;

        // Step 5 — activate account
        ctx.execute_step(activate_account(&data.user_id)).await?;

        Ok(OnboardingResult { customer_id })
    }

    fn max_retries(&self) -> usize { 3 }

    fn timeout(&self) -> Option<Duration> {
        Some(Duration::from_secs(180_000)) // 50h — covers the 48h event wait
    }
}
```
