Zart supports running multiple steps in parallel via `ctx.schedule_step()` and `ctx.wait_all()`. Each scheduled step becomes an independent task in the scheduler — it can run on a different worker node and is fully durable.

## Basic Usage

```rust
use zart::prelude::*;
use async_trait::async_trait;

// Define each step as a ZartStep struct
struct SendEmailStep { email: String }

#[async_trait]
impl ZartStep for SendEmailStep {
    type Output = ();
    fn step_name(&self) -> Cow<'static, str> { Cow::Borrowed("send-email") }
    async fn run(&self, _ctx: StepContext) -> Result<Self::Output, StepError> {
        send_email(&self.email).await
    }
}

struct SetupBillingStep { email: String }

#[async_trait]
impl ZartStep for SetupBillingStep {
    type Output = String;
    fn step_name(&self) -> Cow<'static, str> { Cow::Borrowed("setup-billing") }
    async fn run(&self, _ctx: StepContext) -> Result<Self::Output, StepError> {
        setup_stripe_customer(&self.email).await
    }
}

struct ProvisionResourceStep { user_id: String }

#[async_trait]
impl ZartStep for ProvisionResourceStep {
    type Output = String;
    fn step_name(&self) -> Cow<'static, str> { Cow::Borrowed("provision-resources") }
    async fn run(&self, _ctx: StepContext) -> Result<Self::Output, StepError> {
        provision_aws(&self.user_id).await
    }
}

pub struct OnboardingTask;

#[async_trait]
impl DurableExecution for OnboardingTask {
    type Data   = OnboardingData;
    type Output = ();

    async fn run(
        &self,
        ctx: &mut TaskContext,
        data: OnboardingData,
    ) -> Result<(), TaskError> {
        // Schedule three parallel steps
        let h1 = ctx.schedule_step(SendEmailStep { email: data.email.clone() });
        let h2 = ctx.schedule_step(SetupBillingStep { email: data.email.clone() });
        let h3 = ctx.schedule_step(ProvisionResourceStep { user_id: data.user_id.clone() });

        // Durably wait for all three to complete
        let _results = ctx.wait_all(vec![h1, h2, h3]).await?;

        // This line only runs after all three succeed
        ctx.execute_step(NotifyReadyStep { email: data.email }).await?;

        Ok(())
    }
}
```

## How It Works

Each `ctx.schedule_step()` call:

1. Inserts a new child task into `zart_executions` with a reference back to the parent.
2. Returns a `StepHandle` — a token identifying that scheduled task.
3. The child task can be picked up by **any available worker**, potentially running on a different machine.

`ctx.wait_all(handles)`:

1. Marks the parent execution as **waiting** in the database.
2. Returns the parent to the queue — no worker thread is blocked.
3. When all child tasks complete, the scheduler re-queues the parent.
4. On resumption, `wait_all` returns the collected results.

<Aside type="tip">
`wait_all` is fully durable. If the parent process restarts while waiting, the parent is simply not re-scheduled until all children complete. Child tasks continue running independently.
</Aside>

## Getting Results

Each scheduled step returns its output through the `wait_all` result vector, in the same order as the handles:

```rust
struct SendEmailStep { email: String }
// ... impl ZartStep ...

struct SetupStripeStep { email: String }
// ... impl ZartStep ...

let h1 = ctx.schedule_step(SendEmailStep { email: data.email.clone() });
let h2 = ctx.schedule_step(SetupStripeStep { email: data.email.clone() });

let results = ctx.wait_all(vec![h1, h2]).await?;
// results[0] = output of SendEmailStep
// results[1] = output of SetupStripeStep
```

## Error Handling

If any child task fails (returns `Err`), `wait_all` returns that error. Other in-flight tasks are **not cancelled** — they run to completion (or failure), but their results are not returned to the parent.

```rust
let results = ctx.wait_all(vec![h1, h2, h3]).await;
match results {
    Ok(outputs) => { /* all succeeded */ }
    Err(e) => {
        // At least one child failed
        // The parent can retry — already-completed children won't re-run
        return Err(e);
    }
}
```

<Aside type="note">
Because each child step is independently persisted, re-running the parent after a partial failure will skip already-completed children and only retry the failed ones.
</Aside>

## Dynamic Fan-Out

Schedule a variable number of parallel tasks based on runtime data:

```rust
struct ProcessRegionStep { region: String }

#[async_trait]
impl ZartStep for ProcessRegionStep {
    type Output = RegionResult;

    // Dynamic name — each region gets its own database key.
    fn step_name(&self) -> Cow<'static, str> {
        Cow::Owned(format!("process-{}", self.region))
    }

    async fn run(&self, _ctx: StepContext) -> Result<Self::Output, StepError> {
        process_region(&self.region).await
    }
}

#[async_trait]
impl DurableExecution for ProcessRegionsTask {
    type Data   = MultiRegionJob;
    type Output = Vec<RegionResult>;

    async fn run(
        &self,
        ctx: &mut TaskContext,
        data: MultiRegionJob,
    ) -> Result<Self::Output, TaskError> {
        // Schedule one task per region dynamically
        let handles: Vec<_> = data.regions
            .iter()
            .map(|region| ctx.schedule_step(ProcessRegionStep { region: region.clone() }))
            .collect();

        let raw_results = ctx.wait_all(handles).await?;
        let results: Vec<RegionResult> = raw_results
            .into_iter()
            .map(|r| r.map_err(|e| TaskError::StepFailed { step: "dynamic".into(), source: e }))
            .collect::<Result<_, _>>()?;

        Ok(results)
    }
}
```

## Nesting Parallel Steps

Parallel steps can themselves schedule further parallel steps, creating a tree of concurrent work:

```rust
// Parent: fans out to two pipelines
ctx.schedule_step(PipelineAStep);
ctx.schedule_step(PipelineBStep);

// Inside PipelineA — further parallelism
ctx.schedule_step(PipelineAStep1);
ctx.schedule_step(PipelineAStep2);
```

Each level is fully durable and can span multiple worker nodes.

## Comparison to `ctx.execute_step()`

| | `ctx.execute_step()` | `ctx.schedule_step()` + `wait_all` |
|---|---|---|
| Execution | Sequential, in-process | Parallel, potentially cross-worker |
| Durability | Yes | Yes |
| Result storage | Yes | Yes |
| Best for | Sequential work | Independent, parallelizable work |
| Blocking | Awaits immediately | Awaits all at `wait_all` |
