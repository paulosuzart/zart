Capture variables let you durably record a value in a single expression, without defining a full step. They're perfect for timestamps, environment reads, generated identifiers, or any pure computation that needs to survive process restarts.

## What Problem Do They Solve?

Sometimes you need to capture a value durably, but defining a full step just to return `Utc::now()` feels excessive:

```rust
// Before captures — three definitions just to record timestamps
#[zart_step("capture-started-at")]
async fn capture_started_at(_tag: String, ctx: StepContext) -> Result<String, StepError> {
    Ok(chrono::Utc::now().to_rfc3339())
}

#[zart_step("capture-resumed-at")]
async fn capture_resumed_at(_tag: String, ctx: StepContext) -> Result<String, StepError> {
    Ok(chrono::Utc::now().to_rfc3339())
}

// In the durable handler:
let started_at: String = ctx.execute_step(capture_started_at("started".to_string())).await?;
```

With captures, this collapses to a single line:

```rust
let started_at = zart_capture!(ctx, "started-at", chrono::Utc::now());
```

## How Capture Works

A capture evaluates an expression once, persists the result, and returns the value. On replay after a restart, the cached value is returned — the expression is never re-evaluated.

```rust
use zart::prelude::*;
use zart::zart_capture;

// First run: evaluates the expression, writes to database, returns the value
// Replay: returns the cached value from the database
let started_at = zart_capture!(ctx, "started-at", chrono::Utc::now());
```

The `zart_capture!` macro expands to `ctx.capture("name", || expr).await?`, so you write the expression directly — no closure syntax needed.

## The `ctx.now()` Helper

For the common case of capturing the current UTC timestamp, use the built-in `now()` method:

```rust
let started_at = ctx.now("started-at").await?;

ctx.sleep(Duration::from_secs(5)).await?;

let resumed_at = ctx.now("resumed-at").await?;
```

This is shorthand for `ctx.capture("started-at", chrono::Utc::now).await?`.

## Full Example

```rust
use zart::prelude::*;
use zart::zart_capture;
use serde::{Deserialize, Serialize};
use std::time::Duration;

#[derive(Debug, Clone, Serialize, Deserialize)]
struct SleepInput {
    task_name: String,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
struct SleepOutput {
    task_name: String,
    started_at: String,
    resumed_at: String,
}

struct SleepTask;

#[async_trait::async_trait]
impl DurableExecution for SleepTask {
    type Data = SleepInput;
    type Output = SleepOutput;

    async fn run(
        &self,
        ctx: &mut TaskContext,
        data: Self::Data,
    ) -> Result<Self::Output, TaskError> {
        let started_at = zart_capture!(ctx, "started-at", chrono::Utc::now());

        ctx.sleep(Duration::from_secs(5)).await?;

        let resumed_at = zart_capture!(ctx, "resumed-at", chrono::Utc::now());

        Ok(SleepOutput {
            task_name: data.task_name,
            started_at: started_at.to_rfc3339(),
            resumed_at: resumed_at.to_rfc3339(),
        })
    }
}
```

## When to Use Capture

| Scenario | Use Capture? |
|---|---|
| Recording a timestamp | ✅ Yes |
| Reading an environment variable | ✅ Yes |
| Generating a UUID or correlation ID | ✅ Yes |
| Pure computation (hash, encode, transform) | ✅ Yes |
| HTTP call or database write | ❌ No — use `#[zart_step]` |
| Anything with side effects | ❌ No — use `#[zart_step]` |
| Work that needs retries | ❌ No — use `#[zart_step]` with retry config |

Captures must be **pure** — no side effects, no I/O. The expression may never run again after the first evaluation.

## Step Name Rules

Just like regular steps, each capture must have a **unique, stable name** within an execution. The name is the database key used to find the cached value on replay.

```rust
// ✅ Good — stable, descriptive names
let started_at = zart_capture!(ctx, "started-at", chrono::Utc::now());
let user_tz = zart_capture!(ctx, "user-tz", env::var("TZ").unwrap_or_default());

// ❌ Bad — names that change between runs
let val = zart_capture!(ctx, &format!("value-{}", data.id), compute());
```

If you need to capture a value per iteration in a loop, include the loop index in the name — see [Durable Loops](/rust-api/loops/) for patterns.

## Error Handling

Duplicate capture names return an error — this protects against accidentally reusing a name and getting stale data:

```rust
let a = zart_capture!(ctx, "my-value", 42);
let b = zart_capture!(ctx, "my-value", 99); // StepError: duplicate capture name
```

During step-mode replay, if a capture is not found in the database (e.g., the step name was changed after the execution started), an error is returned. This means captures added to an existing execution should not be renamed or removed.
