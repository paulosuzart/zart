Regular Rust loops are not durable — if your process restarts mid-loop, all progress is lost and the loop starts over. Zart solves this by giving each iteration a unique step name that is persisted to the database.

## The Core Pattern

The key insight is simple: **give each iteration a unique step name**. Zart checks whether a step has already been completed before calling the step logic.

```rust
use zart::prelude::*;
use zart_macros::zart_durable;

#[zart_step("fetch-items")]
async fn fetch_items(ctx: StepContext) -> Result<Vec<Item>, StepError> {
    db.fetch_pending_items().await
}

#[zart_step("process-item")]
async fn process_item(item: &Item, ctx: StepContext) -> Result<(), StepError> {
    processor.handle(item).await
}

#[zart_durable("batch-processor")]
async fn process_batch(
    ctx: &mut TaskContext,
    data: BatchData,
) -> Result<BatchResult, TaskError> {
    // First, fetch items — this step is itself persisted
    let items = ctx.execute_step(fetch_items()).await?;

    // Each iteration gets a unique step name
    for (i, item) in items.iter().enumerate() {
        let step = ProcessItemStep { item: item.clone(), index: i };
        ctx.execute_step(step).await?;
    }

    Ok(BatchResult { processed: items.len() })
}
```

If the process restarts after processing items 0–3, Zart will skip `"process-item-0"` through `"process-item-3"` (returning their stored results immediately) and resume from `"process-item-4"`.

<Aside type="tip">
Always fetch the list of items in a step too. That way the exact same list is used on replay — even if the underlying data changes between attempts.
</Aside>

## Using a plain `for` loop

The `z_durable_loop!` macro was removed. Instead, use a plain `for` loop with `ctx.execute_step()`. Each iteration's step name must be unique:

```rust
use zart::prelude::*;
use zart_macros::zart_durable;

#[zart_step("process-item")]
async fn process_item(item: &Item, ctx: StepContext) -> Result<(), StepError> {
    processor.handle(item).await
}

#[zart_durable("batch-processor")]
async fn process_batch(
    ctx: &mut TaskContext,
    data: BatchData,
) -> Result<(), TaskError> {
    let items = ctx.execute_step(fetch_items()).await?;

    for (i, item) in items.iter().enumerate() {
        let step = ProcessItemStep { item: item.clone(), index: i };
        ctx.execute_step(step).await?;
    }

    Ok(())
}
```

## Polling Loop

Poll an external API until a condition is met. Each poll attempt is a separate durable step:

```rust
use zart::prelude::*;
use zart_macros::{zart_durable, zart_step};

#[zart_step("check-status")]
async fn check_status(attempt: usize, ctx: StepContext) -> Result<Status, StepError> {
    println!("Poll attempt {}", attempt + 1);
    api.check_job_status().await
}

#[zart_durable("job-monitor")]
async fn monitor_job(
    ctx: &mut TaskContext,
    data: MonitorInput,
) -> Result<JobResult, TaskError> {
    for attempt in 0..60 {
        let status = ctx.execute_step(check_status(attempt)).await?;

        match status {
            Status::Completed(result) => return Ok(result),
            Status::Running => {
                tokio::time::sleep(Duration::from_secs(30)).await;
            }
            Status::Failed(err) => {
                return Err(TaskError::StepFailed {
                    step: "check-status".into(),
                    source: err,
                });
            }
        }
    }

    Err(TaskError::custom("job timed out after 60 polls"))
}
```

If the process restarts during the polling loop, completed polls are skipped and the loop resumes from where it left off.

## Transforming Collections

Read items, transform each one, write results — all durable:

```rust
use zart::prelude::*;

#[zart_step("fetch-records")]
async fn fetch_records(ctx: StepContext) -> Result<Vec<Record>, StepError> {
    db.query("SELECT * FROM pending").await
}

#[zart_step("transform-record")]
async fn transform_record(record: &Record, ctx: StepContext) -> Result<Output, StepError> {
    transformer.apply(record).await
}

#[zart_step("write-results")]
async fn write_results(outputs: &[Output], ctx: StepContext) -> Result<(), StepError> {
    db.batch_insert(outputs).await
}

#[zart_durable("data-pipeline")]
async fn run_pipeline(
    ctx: &mut TaskContext,
    data: PipelineInput,
) -> Result<(), TaskError> {
    let records = ctx.execute_step(fetch_records()).await?;

    let mut outputs = Vec::new();
    for (i, record) in records.iter().enumerate() {
        let step = TransformStep { record: record.clone(), index: i };
        let output = ctx.execute_step(step).await?;
        outputs.push(output);
    }

    ctx.execute_step(write_results(&outputs)).await?;
    Ok(())
}
```

## Chunked Processing

For large datasets, process in fixed-size chunks:

```rust
const CHUNK_SIZE: usize = 100;

#[zart_step("get-chunk-count")]
async fn get_chunk_count(total: usize, ctx: StepContext) -> Result<usize, StepError> {
    Ok(total.div_ceil(CHUNK_SIZE))
}

#[zart_step("process-chunk")]
async fn process_chunk(offset: usize, ctx: StepContext) -> Result<(), StepError> {
    let items = db.fetch_items(offset, CHUNK_SIZE).await?;
    processor.process_batch(&items).await
}

#[zart_durable("chunked-processor")]
async fn run_chunked(
    ctx: &mut TaskContext,
    data: BatchInput,
) -> Result<(), TaskError> {
    let total = data.total_items;
    let chunks = ctx.execute_step(get_chunk_count(total)).await?;

    for chunk_idx in 0..chunks {
        let offset = chunk_idx * CHUNK_SIZE;
        ctx.execute_step(ProcessChunkStep { offset, chunk_idx }).await?;
    }

    Ok(())
}
```

## Key Concepts

**Unique step names** — Each loop iteration must have a unique name. Use the loop index: `"process-item-{i}"`, `"chunk-{n}"`, `"poll-{attempt}"`.

**Persisted iteration** — The step result for each iteration is stored in the database. On restart, completed iterations return their cached results instantly.

**Plain `for` loop** — No special macro needed. Use a standard Rust `for` loop with `ctx.execute_step()`. Each step is independently durable.

**Fetch inside a step** — Always wrap the data-fetching call in its own step. This ensures the same data set is used on replay, even if the underlying source changed.
