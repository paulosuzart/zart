The `zart-macros` crate provides proc-macros that transform ordinary Rust async functions into full `DurableExecution` implementations and `ZartStep` step definitions — no trait boilerplate required.

```toml
[dependencies]
zart        = "0.1"
zart-macros = "0.1"
```

## `#[zart_durable]`

Annotate an async function to make it a durable workflow. The macro generates the `DurableExecution` impl, registers the output type, and wires up the `TaskContext`.

```rust
use zart_macros::zart_durable;

#[zart_durable("checkout", timeout = "10m")]
async fn checkout(
    ctx: &mut zart::context::TaskContext,
    order: Order,
) -> Result<Receipt, zart::error::TaskError> {
    // workflow body
    Ok(Receipt { /* ... */ })
}
```

**Parameters:**

| Parameter | Type | Required | Description |
|---|---|---|---|
| `"name"` | string | yes | Task name used when scheduling |
| `timeout` | duration string | no | Wall-clock timeout, e.g. `"30m"`, `"2h"`, `"90s"` |

The function must take `ctx: &mut TaskContext` as the first parameter and return `Result<T, TaskError>` where `T` impls `Serialize`.

## `#[zart_step]`

Turn an async function into a durable step. The macro generates a step struct, implements `ZartStep`, and provides two execution conveniences.

```rust
use zart_macros::zart_step;

#[zart_step("charge-card", retry = "exponential(3, 2s)")]
async fn charge_card(card: &str, amount: i64, ctx: StepContext) -> Result<ChargeResult, StepError> {
    println!("Attempt {}", ctx.current_attempt() + 1);
    stripe.charge(card, amount).await
}
```

**Parameters:**

| Parameter | Type | Required | Description |
|---|---|---|---|
| `"name"` | string | yes | Step name for database tracking |
| `retry` | string | no | Retry config: `"fixed(n, dur)"` or `"exponential(n, dur)"` |
| `timeout` | duration string | no | Per-step timeout, e.g. `"5m"`, `"30s"` |

The last parameter must be named `ctx` and typed `StepContext`. The return type must be `Result<T, StepError>`.

### What the Macro generates

For the `charge_card` example above, the macro produces four items:

1. **`ChargeCardStep` struct** — holds the parameters (`card`, `amount`)
2. **`impl ZartStep`** — with `step_name()`, `retry_config()`, and `run()`
3. **Builder function** — `fn charge_card(card, amount) -> ChargeCardStep` (rewrites the original fn)
4. **`charge_card_step` async fn** — one-call build-and-execute (see option 3 below)

### Three Ways to Call a Step

All three forms are fully equivalent. Choose the one that fits the surrounding code.

**Option 1 — `ctx.execute_step()` (method-first):**

```rust
let charge = ctx.execute_step(charge_card(&card, amount)).await?;
```

The step struct is passed to `ctx`. This is the original pattern, useful when `ctx` is already the focal point of the durable body.

**Option 2 — `.execute(ctx)` (step-first trait method):**

```rust
let charge = charge_card(&card, amount).execute(&mut ctx).await?;
```

The `ExecuteStep` trait is auto-implemented for all `ZartStep` types. Reads naturally when the step is the focus.

**Option 3 — `_step` async function (flat call):**

```rust
let charge = charge_card_step(&mut ctx, &card, amount).await?;
```

The macro generates a `_step`-suffixed async fn that builds and executes in one call. `ctx` is the first parameter, followed by the step's own arguments. No struct or trait visible at the call site.

Steps with references work seamlessly — the macro handles lifetime injection automatically:

```rust
#[zart_step("fetch-data")]
async fn fetch_data(client: &reqwest::Client, url: &str, ctx: StepContext)
    -> Result<String, StepError>
{
    client.get(url).send().await?.text().await
}

// Option 1:
let data = ctx.execute_step(fetch_data(&client, &url)).await?;

// Option 2:
let data = fetch_data(&client, &url).execute(&mut ctx).await?;

// Option 3:
let data = fetch_data_step(&mut ctx, &client, &url).await?;
```

## `z_wait_event!`

Suspend the workflow until an external event is delivered. The process can restart freely; the workflow resumes when the event arrives.

```rust
use zart_macros::z_wait_event;

let approval: ApprovalPayload =
    z_wait_event!("manager-approval", timeout = "24h").await?;

println!("Approved by: {}", approval.reviewer);
```

Returns `Err(TaskError::EventTimeout)` if the timeout expires before the event is delivered.

## `z_durable_loop!`

A loop construct whose iteration counter is persisted. Even if the process restarts mid-loop, already-completed iterations are skipped.

```rust
use zart_macros::{zart_durable, z_durable_loop, zart_step};

#[zart_step("process-item")]
async fn process_item(item: &Item, ctx: StepContext) -> Result<Processed, StepError> {
    processor.handle(item).await
}

#[zart_durable("batch-job")]
async fn batch_job(
    ctx: &mut TaskContext,
    batch: BatchInput,
) -> Result<(), TaskError> {
    for item in &batch.items {
        ctx.execute_step(process_item(item)).await?;
    }
    Ok(())
}
```

## Side-by-Side: Macros vs. Manual

The two styles are fully equivalent. Choose the one that fits your team's preferences.

**With macros (recommended for most workflows):**

```rust
use zart::prelude::*;
use zart_macros::{zart_durable, zart_step};

#[zart_step("send-email")]
async fn send_email(email: &str, ctx: StepContext) -> Result<(), StepError> {
    send_welcome_email(email).await
}

#[zart_step("create-stripe-customer", retry = "exponential(3, 2s)")]
async fn create_customer(email: &str, ctx: StepContext) -> Result<String, StepError> {
    create_stripe_customer(email).await
}

#[zart_durable("onboarding", timeout = "50h")]
async fn onboarding(
    ctx: &mut TaskContext,
    data: OnboardingData,
) -> Result<OnboardingResult, TaskError> {
    // All three forms coexist in the same body:
    send_email_step(&mut ctx, &data.email).await?;                    // flat call
    let customer_id = ctx.execute_step(create_customer(&data.email)).await?; // execute_step
    let _: VerifyPayload = ctx.wait_for_event(
        "email-verified",
        Some(Duration::from_secs(172_800)),
    ).await?;
    Ok(OnboardingResult { customer_id })
}
```

**Manually implementing the traits:**

```rust
use zart::prelude::*;
use async_trait::async_trait;
use std::time::Duration;

struct SendEmailStep { email: String }

#[async_trait]
impl ZartStep for SendEmailStep {
    type Output = ();
    fn step_name(&self) -> Cow<'static, str> { Cow::Borrowed("send-email") }
    async fn run(&self, _ctx: StepContext) -> Result<Self::Output, StepError> {
        send_welcome_email(&self.email).await
    }
}

struct CreateCustomerStep { email: String }

#[async_trait]
impl ZartStep for CreateCustomerStep {
    type Output = String;
    fn step_name(&self) -> Cow<'static, str> { Cow::Borrowed("create-stripe-customer") }
    fn retry_config(&self) -> Option<RetryConfig> {
        Some(RetryConfig::exponential(3, Duration::from_secs(2)))
    }
    async fn run(&self, _ctx: StepContext) -> Result<Self::Output, StepError> {
        create_stripe_customer(&self.email).await
    }
}

pub struct OnboardingTask;

#[async_trait]
impl DurableExecution for OnboardingTask {
    type Data   = OnboardingData;
    type Output = OnboardingResult;

    async fn run(
        &self,
        ctx: &mut TaskContext,
        data: OnboardingData,
    ) -> Result<Self::Output, TaskError> {
        ctx.execute_step(SendEmailStep { email: data.email.clone() }).await?;
        let customer_id: String = ctx.execute_step(CreateCustomerStep { email: data.email.clone() }).await?;
        let _: VerifyPayload = ctx.wait_for_event(
            "email-verified",
            Some(Duration::from_secs(172_800)),
        ).await?;
        Ok(OnboardingResult { customer_id })
    }

    fn max_retries(&self) -> usize { 3 }
    fn timeout(&self) -> Option<Duration> { Some(Duration::from_secs(180_000)) }
}
```

<Aside type="tip">
The macro style generates exactly these trait impls at compile time. You can inspect what it generates by running `cargo expand` on your crate.
</Aside>

## Duration Strings

Both `#[zart_durable]`, `#[zart_step]`, and `z_wait_event!` accept human-readable duration strings:

| String | Duration |
|---|---|
| `"30s"` | 30 seconds |
| `"5m"` | 5 minutes |
| `"2h"` | 2 hours |
| `"1d"` | 1 day |
